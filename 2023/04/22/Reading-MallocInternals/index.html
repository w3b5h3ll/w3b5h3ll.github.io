Reading: MallocInternals

发表于 2023-04-22 更新于 2024-05-27 阅读次数： Valine：

在进入堆溢出的世界前，还是有必要来了解一下Glibc上关于该部分的实现：Ptmalloc。结合MallocInternals和glibc2.23以及glibc2.26(tcache)的源码进行深入了解。对于内存管理的设计，华庭的文章更加值得一读。

原文地址：https://sourceware.org/glibc/wiki/MallocInternals

Malloc总览

GNU C库中的malloc库包含了一些函数，可以用于对在应用程序地址空间分配的内存进行管理。glibc malloc来源于ptmalloc(pthread malloc)，而ptmalloc来源于dlmalloc(Doug Lea malloc)。它是一种“堆”风格的malloc（不同大小的chunk存在于一大块内存区域中），而不是类似于bitmaps和数组或者相同大小的区域等的实现方式。早期时候，每个应用只有一个堆，但glibc的malloc允许有多个堆，每个堆可以在自己的地址空间增长。

下面是本文中的常用术语：

  • Arena

一种在一个或多个thread之间共享的结构，指向了一个或多个heap以及由free chunks组成的链表。分配给每个arena的thread将从arena中的free list进行分配内存。

  • Heap

一块连续的内存区域，它被分成chunks进行内存分配。每个heap只属于一个arena。

  • Chunk

一段小范围的内存块，可以被allocated(分配出去后由应用程序占有)，可以被free(释放后由glibc占有)，或者与相邻的块合并成一个更大的块。每个块存在于heap中并且只属于一个arena。

  • Memory

应用地址空间的一部分。

Chunk

Glibc的malloc是面向chunk的，它把堆空间划分为了不同大小的chunk，每个chunk都包含一些元信息，比如它的大小（通过chunk header中的size获取）、相邻chunk的地址等。当一个chunk被应用所使用时，size是比较关键的信息。当chunk被free时，原来作为应用数据的内存空间被用于其他arena相关的信息，例如链表中的指针，这样可以快速找到并重新使用合适的chunk。在空闲chunk中，可以看到使用了下一个chunk的prev_size字段，该字段在被malloc出去后由于是无效的，所以可以被用来作为上一个chunk的数据区。

在malloc库内部，chunk指针（mchunkptr）并不指向chunk的开始，而是指向上一个chunk的最后一个字段，这表明，只有上一个块是free时，mchunkptr第一个字段才是有效的。

所有的chunk都是8的倍数，size字段的后三位被用作标记位。AMP：

  • A

Allocated Arena，mian arena使用应用空间的heap，而其他的arena使用mmap出来的heap，为了将chunk与heap进行映射对应，需要了解不同的情况。如果该位置为0，那么表面该chunk来自main arena以及main heap，如果为1，表明来自mmap出来的内存空间，并且heap的地址可通过chunk的地址计算出来。

  • M

MMap出来的chunk，chunk是由系统调用mmap分配的，并不是heap的一部分。

  • P

Previous chunk in use，前一个chunk是否在使用。1，表明前一个chunk在使用，同时也说明prev_size字段失效。需要特别注意的是：某些chunk，例如在fastbin里面的chunk，尽管已经是free状态，但是该字段仍然被设置，其真正的意义在于表明前面的chunk不应被考虑合并，起一个优化作用。

为了确保chunk的payload区域足够大来容纳malloc所需要的开销（overhead），一个最小的chunk的大小为4*sizeof(void*)，这个size可能会更大，如果平台需要额外对齐等。

（关于该overload，glibc中有如下说明，说这个开销用来保存size以及状态信息）prev_size不会增加chunk的size为5*sizeof(void*)，因为当chunk足够小，指针bk_nextsize不会被使用。

request size与实际分配的chunk大小转换，需要进行对齐，32位：8字节，64位：16字节

Arenas and Heaps

为了有效管理多线程应用，glibc允许超过一个或更多的内存区域处于活跃状态。不同的线程可以访问不同的arena，而不需要与其他的arena进行交互。main arena对应于应用初始的heap。有一个静态指针指向该arena，每个arena有next指针指向下一个arena，这样多个arena被链接起来了。

每个arena结构中有一个锁用于控制对该arena的访问。需要注意的是：对于fastbins的访问，是原子操作，不需要进行加锁、解锁。互斥锁的争用是多个arena被创建的原因，分配给不同arena的线程不需要互相等待，线程会自动切换到未使用的arena上，如果存在争用。

每个arena从一个或多个heap中获取内存空间。main arena使用程序的初始化heap(.bss段等初始化完后)。其余的arena通过mmap为堆分配内存，随着先前的堆的使用完，继续添加更多的heap。每个arena会跟踪一个特殊的"top" chunk，这是最大的可用chunk，并且指向最近分配的heap。

从arena分配的内存，可以很方便的从初始heap中获取。

在每个arena中，chunk要不就是被使用，要不就是被free。使用中的chunk不被arena所追踪，而free chunk根据其不同的size和历史，被放在不同的list中，以便系统库可以找到合适的chunk去满足内存分配请求。这里list被称作"bins"。

Bins

如下：

  • Fast

小的chunk被存储在特定大小的bins里面。fastbin中的chunk是不会与相邻的chunk进行合并的。（它的逻辑是最小保持最快）fastbin中的chunk会在需要的时候移动到其他的bins里面。其中的chunk存储在单链表数组里，因为这些块大小相同，并不需要访问中间的chunk。

fastbin定义在结构体malloc_state中，如下

可以看到，最大请求size为80/160字节。NFASTBINS为10，总共有10个fasbin链，然而默认情况下，会有如下设置，即默认最大的为64/128。所以一般fastbinY数组后三个数组是不用的，即index为7，8，9未使用。

  • 32bit: [0x10, 0x18, 0x20, …, 0x40]，间隔0x08，共7个

  • 64bit: [0x20, 0x30, 0x40, …, 0x80]，间隔0x10，共7个

在arena的结构中，还有一个特殊的数组bins，其中包含了Unsorted bins、Small bins、以及Large bins。总共有128条双向链表，即128个bin。bin[0]，bin[127]不存在

  • Unsorted

FIFO原则，存放在bins数组的index为1的位置，即bins[1]。unsorted bin可以看作small bins和large bins的一个cache，只有一个双向链表。当chunk最初被释放时，最初被放入unsorted bin中，而分配时，如果该bin中没有合适的chunk，会将其所有的chunk分别加入到所属的bin中，然后再再bin中分配合适的chunk。

注意：初始化时，会将top chunk初始化为unsorted bin，但top chunk并不属于任何一个bin。

  • Small

通用的bins被分割成small bins，每个bin中，chunk的大小相同。small bin存储在bins数组中，共有62个bin，每个bin相差8/16字节。其size<512，其chunk size与index对应关系如下：

Chunk_size = 2 * SIZE_SZ * index，index从2到63

所以，32bit: [0x10, 0x18, …, 0x1f8(504)]，64bit: [0x20, 0x30, …, 0x3f0(1008)]。

  • Large

Large bin是其中chunk大于等于512的bin，有63个，index从64到126。每个bin中的chunk大小不是固定的等差。将其分为了6组。32位系统下的情况如下：

32: 64，该组内公差为64，该组有32个bin

16: 512，同上

8: 4096

4: 32768

2: 262144

1: left

对于samll bin来说，你可以挑选第一个chunk并使用它，而对于large bin来说，需要找到最合适的chunk，还有可能需要将其分割为两个chunk。（一个供使用，另一个为remainder）

large bin中的chunk是从大到小排序的，chunk结构中的*fd_nextsize与*bk_nextsize指针开始发挥作用，形成了一个double list，这样可以快速找到合适的足够大的chunk。注意：如果存在多个不同size的合适的块，一般选择第二个。（第一个可能还需要进行分割）

Tcache

Thread Local Cache，tcache在glibc2.26时被引入。在多线程应用中，每个thread有一个thread-local变量用来标记它最近使用的arena，如果该arena被使用，那么该thread将阻塞并等待该arean被释放。如果该线程从未使用过arena，那么，它可能创建一个新的arena，或者从全局列表中挑选一个，或重用一个没有被使用的arena。

每个线程都有一个tcache，该cache包含了许多chunk，可以在不对arena加锁的情况下进行访问。（提高堆管理性能）这些chunks被存储在包含singly-linked lists的数组中。其指针指向了chunk的user area区域，而不是chunk header。每个bin只包含一个size大小的chunk。tcache对于每个bin中的chunk是有限制的，tcache_cout。如果请求的size在tcache中找不到，那么下一个更大的sized chunk将不被使用，这会造成内部碎片。相反，它会回退到正常的malloc routines，对thread的arena加锁，并开始工作。

默认情况下，tchache中，每个bin上最大的chunk数目:7

  • tcache_entry，每个bin的结构

Malloc Algorithm

对malloc的简单工作流程说明

  • 如果在tcache中有合适的chunk（精确匹配），直接返回给调用者。不会尝试使用来自large-sized bin中的chunk。

  • 如果请求足够大，使用mmap()直接向操作系统申请内存。注意：对于mmap来说，它是有阈值的，即请求在这个阈值时，才调用mmap，否则使用heap管理。这个阈值是动态的，除非修改了M_MMAP_THRESHOLD，并且对于一次可以有多少这样的映射也有限制。

M_MMAP_THRESHOLD is the request size threshold for using mmap()
to service a request. Requests of at least this size that cannot
be allocated using already-existing space will be serviced via mmap.
(If enough normal freed space already exists it is used instead.)

  • 如果在fastbin中有合适的chunk，使用它。如果还有另外合适的chunk，那么对tcache进行填充。

  • 如果在smallbin中有合适的chunk，使用它。可能也对tcache进行预先填充。

  • 如果请求属于”large”，先对fastbins中的所有放入unsorted bin中，并进行合并

  • 将unsorted bin中的chunk取出来，将它们移动至对应的small/large bins中，进行合并。（这是唯一将chunks放入small/large bin中的时机）如果有合适size的chunk存在，使用它。

  • 如果请求属于”large”，查找合适的large bin，返回合适的chunk。

  • 如果在fastbins中仍然存在chunks，合并并重复前两个步骤。（如果是”small request”，并且unsorted bin中只有last remainder尝试使用last remainder）

  • 分割top chunk，可能会预先增大top chunk。

对于强调对齐malloc来说，例如valloc、pvalloc、memalign，一个overly-large的chunk通过上面的流程获取，并分割为更多的chunk，大多数的chunk会对齐的很合适，该步骤之前和之后的部分被返回给unsorted list。

Free Alogorithm

一般来说，释放内存并不会将其返回给操作系统供其他应用去使用。free()调用会标记该chunk可以被该应用重新使用，但在操作系统的视角来看，它仍然属于该应用。然而，如果top chunk变得足够大，其中的某些内存可能会被取消映射，并返还给操作系统。

free的流程如下：

  • 如果tcache未被填满，那就将chunk放入tcache并返回

  • 如果chunk足够小，将其放入合适的fastbin中

  • 如果chunk是mmap出来的，munmmap它

  • 如果chunk与free chunk相邻，合并它

  • 将chunk放入unsorted bin，除非它当前是”top” chunk

  • 如果chunk足够大，合并fastbin中所有的chunk，并检查top chunk是否足够大去给系统返还一些内存。（注意：这个过程可能会被推迟，出于性能的原因，在malloc或其他call期间发生）

Detecting heap corruption

malloc子系统会检测是否存在heap corruption。有些检测的结果是一致的，比如：给free函数传递一个不对齐的指针。然而，大多数检查方法都是启发式的，可能会被看起来真实的完全虚假的数据块所欺骗。（比如检查fd/bk相连的chunk），如果是这样的话，heap corruption就不会被发现，也不会被报告。

启发式的检查方法是指使用经验和规则来判断数据块是否真实有效，而不是依赖于严格的算法和逻辑。由于这种方法的不确定性和局限性，有些人可能会试图通过制造看起来真实的虚假数据块来欺骗检查系统。因此，尽管检查方法可以提高数据的可信度，但它们并不能保证完全准确和可靠。

通常使用malloc_printerr来处理corruption，更进一步的检查使用assert来完成，然而被禁用来在构建glibc的过程中使用-DNDEBUG参数。在当前的 glibc 中，这两种检查都通过调用__libc_message 来终止进程，后者最终调用abort。非常老的glibc版本中，支持在出现heap corruption的情况下继续运行，但是这种支持已经被移除了。
